# Deploy Pipeline Runbook

This runbook covers release orchestration, environment promotion, and rollback procedures for the Core Deployment Pipeline.

## Release Orchestration
Our deployment pipeline is orchestrated using GitLab CI/CD runners. Code merged to the `main` branch is automatically compiled, tested, and pushed to our container registry.

## Promotion Steps
1. **Staging Review**: Code must pass automated integration suites in the staging environment.
2. **Production Gate**: Production releases require manual approval in the release dashboard.
3. **Canary Rollout**: We deploy to 5% of traffic initially, monitoring error rates via Prometheus.

## Rollback Procedures
In the event of a critical region-level outage or elevated 5xx error codes (>1% of traffic):
1. **Manual Rollback Action**: Click "Rollback to Last Stable" in the GitLab pipeline UI.
2. **DNS Cutover**: If the CI runner hangs, immediately execute the AWS Route53 DNS cutover scripts to route traffic away from the affected US-East-1 region to US-West-2.
3. **Build Cache Cleansing**: The build cache on the CI runners has known quirks. If build cache invalidation hangs, run `docker builder prune -f` manually on target runner VMs.
